# C 端"公司安不安全"输出事实召回画像，绝不给安全判决 / 评分

Status: accepted
Date: 2026-07-10

## 决定

C 端"这品牌 / 公司安不安全"的回答是一个**事实性召回画像（Recall Profile）**——按 Parent Group 聚合，给出严重度（Class I/II/III）、状态（Ongoing / Terminated）、近期召回，配一句**有边界的解读**，全部可下钻到 `recall_number`。**绝不输出"安全 / 不安全"二元判决或风险评分。** 术语见 [CONTEXT.md](../../CONTEXT.md)。

## 为什么

FDA 召回数据**不测量"公司安全度"**。实测：`Pfizer`+`Hospira` 有 308 条召回（多为已 `Terminated` 的 Class II），而小配药房 `Abrams Royal Pharmacy` 仅 3 条、却**全是 Class I**（可能致命）。按裸计数排名会把大厂误判成"最不安全"。而且一条召回意味着**问题被发现并纠正了**（全库 83% 已 `Terminated`）；**零召回 ≠ 安全**（可能只是小 / 新 / 没人盯）。"X 安不安全"是带法律 / 责任重量的判断，我们既无资质、也无数据下此结论。

## 取舍

- **给判决 / 评分**：C 端爱看、直接，但误导 + 有法律风险，且 FDA 数据支撑不了。
- **给事实画像（本决定）**：诚实、可审计、可辩护；代价是要用严重度 / 近期 / 在进行中等信号帮用户抓重点，而不是甩一堆数字或一个假结论。

## 后果

- 答案层**永远不产出**单一 `safe` / `risky` 标签或风险分数。
- 负结果表述为"FDA 未找到相关召回"，**不是"安全"**。
- 画像信号复用 Path 1 的 `count_by` / `trend`（严重度、状态、时间）。
